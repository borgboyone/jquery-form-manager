jQuery Form Manager

jQuery plugin providing an extensive base framework and toolset for working with web page inputs, either individually or as a group.

Selectors

Input Selectors are assembled using a combination of filter and aggregate directives. Filtering directives fall into two categories, those that examine properties, where the underlying property of the entity can have any value, and those that examine dichotomies, where the underlying property of the entity can have only two values (usually true or false). Aggregation directives on the other hand, dictate how filtered collections are combined into new collections. They consist of common operations used in the merging of sets.

Properties

Properties must appear as key, value pairs delimited by an operator and enclosed in square brackets (‘[’, ’]’) with multiple pairs separated by commas. Currently, only the equal operation (“=”) is supported.

Valid keys consist of:

  • "name": Inputs are filtered based on the value returned by theName
  • "valid": Inputs are filtered based on the value returned by validationStatus
  • "type": Inputs are filtered based on the Widget name (not fullName) of either the base widget or parent widgets

Dichotomies

Dichotomies must be prefixed with either ‘:’ or ‘;’ and may appear alone or in conjunction with an attribute. Inputs with dichotomies prefixed with ‘:’ are kept only if the underlying method evaluates to true, while those prefixed with ‘;’ are only kept if the underlying method evaluates to false. If the Dichotomy is part of an attribute selector, it must appear after the attribute declaration.

Valid Dichotomies consist of:

  • "hasChanged": Inputs are filtered based on the value returned by hasChanged
  • "isValid": Inputs are filtered based on the value returned by isValid
  • "isDisabled": Inputs are filtered based on the value returned by isDisabled

Aggregators

Aggregators are used to specify how groups of entities that have undergone filtering are combined. The type of aggregation that is performed on the groups is determined exclusively by the operator. The following aggregators are supported:

  • “&” Intersect: Filtered Inputs are combined with only those Inputs that appear on both sides returned in the aggregate.
  • “|” Merge: Filtered Inputs are combined with Inputs on both sides returned in the aggregate.
  • “-” Exclude: Filtered Inputs are excluded from all Inputs under the Form. (Technically evaluates as "(*-filtered)" where * indicates all Inputs.)

Grouping

Parenthesises are used to group entities through aggregation operations. They are mainly suited to cases where the evaluation of aggregators is unclear or the default precedence would result in an undesired outcome.

  • “(…)” Used exclusively to determine the evaluation order of aggregation operations.

Caveats

Here are some common mistakes that are made with Input selectors with the correct method provided where applicable:

  • “(…):hasChanged” is not acceptable as a selector. Property and dichotomy filters are only valid at what would be considered the node level of each evaluation branch.
  • "[name=name][name=address]" is not acceptable as an aggregate selector. Use '|' between selectors for these situations.
  • "[name=name][type=type]" is not acceptable as a filter selector. Either use '&' between the property selectors or combine the property name, value pairs: [name=name,type=type].
  • ":hasChanged-:isValid" is not acceptable as a selector. The exclusion operator '-' performs exclusion against all inputs. Use ":hasChanged;isValid" instead.
